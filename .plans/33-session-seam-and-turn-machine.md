# Session Seam Narrowing, Turn Machine, Run-State Codec, ModelTurn Normalization

Deepening refactor from the 2026-07-01 architecture review, design settled via grilling; amended after multi-review v1. Four changes, strictly ordered. Candidate 5 from that review (naming the tool-result metadata contract) is deferred and out of scope.

## Decisions

- `RequestConstants` is a frozen dataclass in `providers.py` holding `instructions`, `tools`, `metadata`, `structured_output`. Built once per run and passed as one positional parameter to every `ModelSession` method. Build point: after `_prepare_run_start` returns — MCP connect appends tools and `system_instructions()` folds tool instructions in (`structured_instructions` adds prompted-schema instructions), so building at `RunContext` construction would drop them. Constants are built from `run_metadata` (the normalized copy), which is content-identical across all three entry paths. The toolset is frozen for the run; mid-run `add_tool` no longer affects in-flight runs (no callers rely on it; all in-repo `add_tool` calls happen before `run()`).
- `notices` stay a per-request keyword argument — they are not constant.
- `ModelSession` shrinks to three request methods plus `dump_state`: `start(prompt, constants, *, previous_response_id=None, notices=None)`, `continue_with_tools(outputs, constants, *, notices=None)`, `continue_with_user_text(text, constants, *, notices=None)`. The byte-identical `continue_with_user_message`/`continue_with_user_prompt` pairs collapse into `continue_with_user_text`; providers receive no request-kind parameter (runtime keeps `ModelRequestKind` for events/tracing). Resume replay stays state-driven (OpenAI `_pending_replay`; Anthropic/OpenRouter `_resume_entries` + `_apply_resume`), so the collapse is behavior-preserving.
- Hard break on the protocol, no deprecation shims. `Model.new_session()`/`resume_session(state)` are unchanged; constants never bind at session construction, so RESUME-5's live-instructions re-injection stays explicit on the resume call.
- `parallel_llm.py` is a consumer of both the session seam and the classifier: its `session.start(...)` call moves to the new signature, it builds its own `RequestConstants` (it has no `RunContext`; metadata `None`), and its `OutputTurnDecision`/`resolve_turn_output` imports move to `turns`.
- New module `thinharness/turns.py` owns the turn state machine: `OutputTurnDecision` and `resolve_turn_output` move in from `output.py`; `advance_until_terminal` owns the loop, explicit dispatch on all five decision kinds (including `continue`), the retry-budget spend, and first-turn production for all three entry paths (start / resume / approval-resume, including the approved-tool-batch replay). It returns the terminal `HarnessResult` — `run_ctx.finalize`/`pause_for_approval` are called inside. Also moving from `core.py`: `_execute_tool_turn`, `_approval_required_calls`, `_validate_approval_resume_tools`, `_reject_approval_call`; `_structured_retry_message` moves with `resolve_turn_output`. Session creation stays in core: core creates or resumes the session before the call, and `advance_until_terminal(start, session, constants, harness, run_ctx, tool_executor)` receives it with explicit collaborators — the inline `agent_span` annotation on `new_session()` failure is dropped (today that failure is recorded twice, once inline and once via `_classify_run_failure`; nothing pins the double record, so it collapses to the single classify-path record). `output.py` keeps `OutputSchema`, `resolve_output_schema_for_model`, `structured_instructions`, `structured_retry_prompt`.
- Exception ownership: the `try/except CancelledError` + `_classify_run_failure` block stays in `core.py` wrapping the single `advance_until_terminal` call; turns.py raises, core classifies, `agent_span` never enters turns.py.
- Classification timing: `RunContext.advance_model` keeps calling `resolve_turn_output` (imported from `turns`) so the decision is resolved inside the still-open model span — the finalized-mode span attributes (`thinharness.output.mode`, `gen_ai.output.finalized`) are set before the span closes, which the local tracer requires. No import cycle: turns.py imports runtime/core types under `TYPE_CHECKING` only.
- Retry-budget ordering is preserved, not changed, and the spend is split across two owners on purpose: the budget check (`retry_or_fail`) runs at dispatch inside `advance_until_terminal`; the `usage.output_retries` increment stays in `advance_model`, after `check_model_limit` admission. `test_retry_not_counted_when_model_limit_blocks_corrective_request` encodes this — a retry blocked by the model-request limit is not counted. Locality comes from co-location in one module, not reordering.
- Loop invariants to preserve (and pin in unit tests): `record_response` runs at loop top before dispatch, so the terminal turn's and the approval-paused turn's raw responses are both recorded and the approval envelope includes the pausing turn; approval-batch replay does not re-run `check_tool_limit` or re-count `tool_calls` (counted at pause time), and `cancelled_tool_calls` accounting for hook-cancelled approved calls is preserved.
- `TurnDriver` is deleted. `core.py` does run-scoped setup (session creation, approval-envelope validation, stream/tracing wiring) and makes one call into `turns.py`.
- `RunContext` keeps per-request ceremony, usage, and terminal bookkeeping; it gains `record_response` and a model-message emission method that takes the finalized mode as an argument — the `ModelTurn.finalized_output_mode` post-hoc mutation is removed.
- Run-state codec lives with the types in `types.py`: `RunUsage.to_json()`/`RunUsage.from_json(data, label=...)` and `LimitNoticeKey` encode/decode. `from_json` preserves current validation semantics exactly: the four counter fields stay required with today's type checks, `tool_retries` stays optional defaulting to `{}`, unknown keys stay ignored, and error messages keep their current full text (label prefix plus field-name substrings, e.g. "field 'usage' has wrong type"). The `input_tokens`/`output_tokens` fields land in change C together with the codec (dataclass default 0; default 0 when absent from JSON; present-but-wrong-type rejected), so the codec is written once against its final key set — change D only accumulates. `approvals.py` deletes `_parse_usage`/`_parse_limit_notice_keys` and delegates. Approval envelope stays version 1; provider transcript v3 is out of scope.
- `ModelTurn` gains normalized `usage: TokenUsage | None` (frozen dataclass; fields `input_tokens: int | None`, `output_tokens: int | None` to tolerate partial usage), `finish_reason: str | None`, and `response_model: str | None` — native strings, no invented taxonomy; richer extras stay in `raw`. New per-provider extractors are written in `providers.py` by relocating the extraction logic from tracing's `_usage_attributes`/`_finish_reasons`/`_response_model` (the existing `_extract_*` helpers cover only text/tool-calls/reasoning); extraction tolerates missing `usage`/`model`/finish keys (`None`, matching today's `.get` behavior), handles both key styles (`input_tokens`/`output_tokens` vs `prompt_tokens`/`completion_tokens`; `stop_reason` vs `choices[].finish_reason`), and preserves today's precedence (`raw["model"]` before `choices[0].model`; `stop_reason`, then top-level `finish_reason`, then `choices[].finish_reason`). Multi-choice narrowing is accepted: normalized `finish_reason` covers `choices[0]` only, matching what the harness consumes.
- Tracing reads the normalized fields; when they are `None` (custom `Model` implementations — a public extension point), it falls back to best-effort raw extraction via a helper exported from `providers.py`, so custom models keep their `gen_ai.usage.*` span attributes. Trace shape is pinned: `gen_ai.response.finish_reasons` stays a list (wrapping the single normalized reason); `gen_ai.usage.total_tokens` follows one rule in both the normalized and fallback paths — raw `total_tokens` passes through when present, otherwise computed as input+output only when both are non-None (Anthropic spans gain it; partial usage yields no total rather than an error).
- `RunUsage` gains `input_tokens`/`output_tokens` run totals accumulated per request, surfaced on `HarnessResult.usage`.
- Metadata truthiness: providers guard payloads with `if metadata:`, which makes `None` vs `{}` equivalent — keep those guards as-is when touching signatures.

## Changes, in order

### A. Narrow the ModelSession seam

- `providers.py`: add `RequestConstants`; protocol and all three sessions go to 3 request methods + constants param; delete the twin user-text duplicates (~60 lines).
- `tools/parallel_llm.py`: new `start` signature; builds its own `RequestConstants`.
- Test blast radius (hand-rolled sessions and direct calls, all updated in this change): `tests/fakes.py`, `test_providers.py` (direct session-method calls with old kwargs — a rewrite, not a rename), `test_structured_output.py`, `test_resume.py`, `test_mcp.py`, `test_parallel_llm.py`, `test_hooks.py`, `test_streaming.py`, `test_tool_retry.py`, `test_harness.py`. `ScriptedSession` can no longer branch on method name for resume-vs-correction: it treats a user-text request as a resume only when it is the first request of any kind on the session — a correction following a resume or approval-resume therefore returns `continue_turn` (no current test exercises that path; change B adds one). The fake keeps recording `(method, notices)` tuples under the collapsed method name for notice assertions, and resume-vs-correction assertions move to `ModelRequestStartedEvent.request_kind`. `on_start`/`on_continue` callback positional shapes are preserved by unpacking constants inside the fake, so `test_subagents.py`, `test_approvals.py`, `test_tracing.py`, and `test_bash_tool.py` need no edits.
- `runtime.py`: `TurnDriver` loses the kwarg re-binding, keeps thin (session-method, request_kind) pairings until change B.
- `thinharness/__init__.py`: export `RequestConstants` in `__all__` (it appears in the public `ModelSession` signature); add an import test and a direct test pinning the toolset freeze (a tool added mid-run does not appear in later requests' schemas).
- Verify: full pytest suite (`test_resume`, `test_reasoning_fidelity`, `test_streaming`, `test_structured_output`, `test_parallel_llm` are the sensitive ones), `uv run pyright`, ruff.

### B. Turn machine

- New `thinharness/turns.py` as decided above; move `OutputTurnDecision`/`resolve_turn_output` from `output.py`; delete `TurnDriver`; `core._run_streaming` loop, `_start_or_resume_turn`, `_resume_approval_batch`, `_execute_tool_turn`, `_approval_required_calls`, `_validate_approval_resume_tools`, `_reject_approval_call` move in; `RunContext` gains `record_response`/model-message emission; remove `ModelTurn.finalized_output_mode`.
- Import updates: `core.py` (`OutputTurnDecision` from `.output`, `TurnDriver` from `.runtime`), `tools/parallel_llm.py:21`, `tests/test_structured_output.py:31`.
- New unit tests: feed `(turn, decision)` pairs to the dispatch and assert the next action without a full `Harness` (the machine takes explicit collaborators, so a minimal fake harness/RunContext suffices) — all five kinds explicitly; retry-budget ordering including the model-limit gate (blocked corrective request not counted); approval pause; a correction following a resume on the same session; `ModelMessageEvent.finalized_output_mode` still populated on finalized turns; the loop invariants (every turn including terminal and paused appears exactly once in `responses`; approval-batch resume does not double-count `tool_calls`; `cancelled_tool_calls` accounting preserved).
- Verify: new unit tests plus full suite; `uv run pyright`; behavior unchanged end-to-end.

### C. Run-state codec

- `types.py`: validating codecs on `RunUsage` (with `label` plumbing for error-message compatibility) and `LimitNoticeKey`; `approvals.py` delegates; `build_approval_envelope`'s `asdict(usage)` becomes `to_json()`.
- New invalid-input tests in `test_approvals.py` (none exist today), asserting the exact current error strings captured before the refactor: missing/wrong-type counter fields raise; omitted `tool_retries` defaults to `{}`; malformed `tool_retries` raises; unknown keys are ignored (no raise); missing token keys default to 0; present-but-wrong-type token keys rejected; `to_json`/`from_json` round-trip equality.
- Verify: `test_approvals`, `test_resume`.

### D. ModelTurn normalization and token totals

- `providers.py`: `TokenUsage`; per-provider extractors (relocated tracing logic); best-effort raw-extraction helper exported for tracing's custom-model fallback.
- `tracing.py`: helpers become field reads with the raw fallback; trace shape pinned as decided.
- `runtime.py`: accumulate totals per request onto the change-C `RunUsage` fields, surfaced on `HarnessResult.usage` (pre-D envelopes resume because the codec already defaults the token keys).
- `thinharness/__init__.py`: export `TokenUsage` in `__all__`.
- New tests: per-provider `TokenUsage`/finish/model extraction, including partial usage (only one of input/output present → no `total_tokens`, no error) and OpenRouter `gen_ai.response.model` reading top-level `model` first; tracing assertions pinning `gen_ai.usage.input_tokens`/`output_tokens`/`total_tokens`, `gen_ai.response.model`, and `gen_ai.response.finish_reasons` list shape (nothing pins these today); a custom `Model` returning `ModelTurn` with `usage=None` keeps its `gen_ai.usage.*` attributes via the raw fallback; `HarnessResult.usage` totals equal the per-turn sum across a multi-turn run including a retry turn and an approval-resume turn.
- Verify: `test_tracing`, `test_providers`, `test_approvals` (old-envelope tolerance), full suite.
