Pass 1 end state

Core asks output.py for one turn decision

Files involved

  • thinharness/output.py gains the structured-turn decision module.
  • thinharness/core.py stops spelling out every output mode branch inline.
  • thinharness/tools/parallel_llm.py can later reuse the same decision shape for one-shot calls.
  • tests/test_structured_output.py becomes the main behavior safety net.

Likely functions

  • Add a small result type such as OutputTurnDecision.
  • Add one resolver function in output.py, for example resolve_turn_output(...).
  • Retire or shrink Harness._finalized_output_mode_for_turn.
  • Keep schema building and validation inside OutputSchema.

Flow after the pass

  1. Harness.run receives a ModelTurn.
  2. It calls the output module with the turn, schema, retry count, and retry budget.
  3. The output module returns one decision: final value, corrective user message, corrective tool output, unexpected model behavior, or continue to normal tools.
  4. core.py performs the side effect implied by that decision: finalize, advance the model with a correction, raise, or continue.
  5. Tracing reads the decision's finalized mode instead of doing separate validation.

What should be easier to read

The main loop should stop containing separate branches for text, tool, native, and prompted output. It should read like: resolve output decision, act on decision, otherwise execute tool calls.

One helper owns one model-requested tool call

Files involved

  • A new internal module or section owns one-call lifecycle policy.
  • thinharness/hooks.py remains the hook dispatch module.
  • thinharness/tracing.py remains the trace annotation module.

Likely functions

  • Keep Harness._execute_tool_batch as the batch module.
  • Replace most of Harness._traced_call_output with a call into a deeper one-call module.
  • Return ToolCallExecution with output, cancellation, retry kind, parsed output, and trace metadata.
  • Move MCP and subagent trace attribution out of the middle of the call path.

Flow after the pass

  1. _execute_tool_batch decides sequential vs concurrent execution.
  2. For each call, the one-call module fires before_tool_call.
  3. If not cancelled, it invokes the tool through the existing tool interface.
  4. It parses output, captures retry kind before after-hooks, fires after_tool_call, then refreshes parsed output.
  5. It applies trace attributes from a small metadata object instead of open-coded conditionals.
  6. The batch module preserves order and builds provider ToolOutput values.

What should be easier to read

The tool path should no longer make the reader hold hook cancellation, retry budgeting, parsed output refresh, subagent metadata, MCP metadata, and trace result capture in one function.

Problem. `Harness.run` is valuable but its implementation has low locality because many concerns share closure and instance state.

Solution. Move per-run mutable state and terminal bookkeeping into an internal run-context module.

  • locality: stop rules central
  • leverage: run_end stays once
  • interface: no metadata global

Pass 2 end state, part A

Harness.run has an explicit per-run state object

Files involved

  • thinharness/core.py owns the new internal run-context type.
  • thinharness/hooks.py keeps the public hook context types unchanged.
  • tests/test_hooks.py, tests/test_resume.py, and tests/test_harness.py verify lifecycle behavior.

Likely functions

  • Add an internal dataclass such as RunContext.
  • Move fire_run_end_once, check_model_limit, check_tool_limit, and finalize into methods.
  • Pass run metadata explicitly to tool execution instead of relying on self._current_run_metadata.
  • Keep HarnessResult public shape unchanged.

Flow after the pass

  1. Harness.run creates RunContext with prompt, metadata, usage, responses, tool records, and tracer.
  2. The run context fires run_start, tracks terminal errors, and guarantees run_end once.
  3. The main loop calls context methods for limit checks and final result construction.
  4. Tool execution receives the run context or the fields it needs, so metadata does not live on the harness instance during the run.
  5. The outer finally still resets _running and releases transient state.

What should be easier to read

Harness.run should look like orchestration rather than bookkeeping: start run, advance model, resolve output, execute tools, continue, finalize.
